# 贡献指南

感谢你有兴趣为 book-to-skill 贡献！

## 贡献方式

### 1. 认领书单中的书

在 [README 书单](README.md) 中找一本你想拆的书，开一个 Issue 说「我来拆这本」，然后开始工作。

### 2. 提议新书

开 Issue，说明：
- 书名和作者
- 为什么这本书适合拆成 Skill（它解决什么具体问题？）
- 你愿意自己拆，还是希望别人来？

### 3. 改进现有 Skill

发现案例不准确、操作步骤不清晰、缺少边界情况？直接提 PR。

## Skill 编写规范

### 目录结构

```
skills/
└── your-skill-name/
    ├── SKILL.md           # 主文件（必须）
    └── references/        # 案例库、参考资料（可选）
        └── cases.md
```

### SKILL.md 格式

```markdown
---
name: your-skill-name
description: 一句话说明触发场景。用户说什么话时应该触发这个 Skill。
---

# Skill 标题

## 任务目标

一段话说明这个 Skill 解决什么问题。

## 操作步骤

### 第一步：...
### 第二步：...

## 注意事项

- 边界情况和常见误用

## 使用示例

### 示例 1：场景名
- **场景**：...
- **操作**：...
```

### 编写原则

1. **一个 Skill 解决一个问题**
   - 好：「用实验替代预测」
   - 坏：「如何做好所有决策」

2. **操作步骤要具体**
   - 好：「问用户：这个选择必须具备怎样的条件，才能成为正确的答案？」
   - 坏：「引导用户思考」

3. **必须有真实案例**
   - 每个核心概念至少配一个案例
   - 案例格式：人物/组织 → 情境 → 做法 → 结果 → 启示
   - 案例放在 `references/cases.md` 中，SKILL.md 里引用关键案例

4. **主 Skill + 子 Skill 结构**
   - 如果一本书的内容太多，拆成一个主 Skill（路由）+ 多个子 Skill（具体操作）
   - 主 Skill 负责判断用户场景，路由到对应子 Skill
   - 参考 `making-decisions`（主）→ `reality-testing-decisions`（子）的结构

5. **中文为主，术语保留英文**
   - Skill 内容用中文
   - 代码标识、专有名词保留英文

## 拆书流程

推荐使用 `/book-to-skill` Skill 来辅助拆书，它会引导你完成以下步骤：

1. 通读全书，识别核心框架
2. 提取可操作的步骤（不是理论总结）
3. 收集案例，按框架分类
4. 设计 Skill 结构（主 Skill + 子 Skill）
5. 编写 SKILL.md 和 references
6. 自测：用真实决策场景测试 Skill 是否好用

## 提交 PR

1. Fork 本仓库
2. 创建分支：`git checkout -b book/书名英文`
3. 按规范编写 Skill
4. 提交 PR，说明拆了哪本书、包含哪些 Skill
